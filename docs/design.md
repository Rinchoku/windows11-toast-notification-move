# 設計思想

## 目的

Windows 11 のトースト通知はデフォルトで右下に表示されるが、OS の設定で変更する手段がない。本ツールはその制約を最小限のリソースで回避することを目的とする。

## 設計方針

### 軽量性を最優先する

TopNotify など既存のツールは GUI・アニメーション・自動更新機能を持ち、常時 CPU・メモリを消費する。本ツールはウィンドウ位置の変更という単一の責務に徹し、アイドル時の CPU 使用率をほぼ 0% に抑える。

### 依存を最小化する

- .NET Framework 4.x は Windows に標準搭載されており、追加インストール不要
- .NET SDK・NuGet パッケージ・サードパーティライブラリは一切使用しない
- 単一の `.cs` ファイルと `csc.exe` だけでビルドできる

### OS の仕組みに乗っかる

ポーリング（定期的な状態確認）を避け、Windows のイベント通知機構（WinEvent フック）を使う。イベントが発生しない間はスレッドがメッセージループで待機するだけであり、リソースをほぼ消費しない。

## 技術的な判断

### WinEvent フックの選択

通知ウィンドウの出現を検知する方法として以下を検討した。

| 方法 | 採否 | 理由 |
|---|---|---|
| ポーリング（EnumWindows を定期実行） | 不採用 | CPU を継続的に消費する |
| WinEvent フック（SetWinEventHook） | **採用** | イベント駆動でアイドル時 CPU ほぼ 0% |
| Windows Notification Listener API | 不採用 | UWP アプリ登録が必要で複雑 |

`EVENT_OBJECT_SHOW`（ウィンドウ出現）と `EVENT_OBJECT_LOCATIONCHANGE`（位置変更）の両方を監視することで、通知アニメーションの初期フレームから追跡できる。

### 繰り返し移動による対抗

`SetWindowPos` で一度移動しても、Windows の通知アニメーションシステムが毎フレーム位置を上書きする。そのため、ウィンドウが目標位置（右上）に安定するまで 30ms ごとに繰り返し移動する。

安定の判定は「連続 5 フレーム目標位置に留まった」とし、それ以降は移動を停止する。これにより閉じるアニメーション（上から消える）との干渉も避けられる。

### hwnd 管理の非採用

当初、移動済みの hwnd を `HashSet` で管理し再移動を防ぐ実装を試みた。しかし Windows が通知ウィンドウの hwnd を再利用するため、2回目以降の通知が移動されなくなる問題が発生した。最終的にループ内の安定チェックだけで制御し、hwnd の追跡は行わない設計とした。

### DPI 対応

`SetProcessDPIAware()` を呼び出し、物理ピクセル座標で一貫して扱う。これにより 2x スケール（例: 2880x1800）の環境でも正確な位置計算ができる。

### `winexe` ターゲット

コンソールウィンドウが表示されないよう `/target:winexe` でビルドする。デバッグ出力はコンソールではなく exe と同じディレクトリの `debug.log` に書き出す。

## 既知の制約

- **Windows アップデートへの非耐性** — 対象ウィンドウのプロセス名・クラス名（`ShellExperienceHost` / `Windows.UI.Core.CoreWindow`）が変わると動作しなくなる。その場合は `--debug` モードでクラス名を特定し、`IsNotificationWindow` を修正してビルドし直す
- **プライマリモニターのみ対応** — マルチモニター環境では `GetSystemMetrics(SM_CXSCREEN)` がプライマリモニターの解像度を返すため、セカンダリモニターの通知は移動できない
- **管理者権限不要** — タスクスケジューラへの登録も `RunLevel Limited`（通常権限）で動作する
